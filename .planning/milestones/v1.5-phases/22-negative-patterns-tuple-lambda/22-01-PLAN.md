<task id="1" title="Add negative integer pattern to parser">
**Parser.fsy:** Add a new Pattern rule for negative numbers:

```
| MINUS NUMBER  { ConstPat(IntConst(-$2), ruleSpan parseState 1 2) }
```

Add this in the Pattern rules near the existing `| NUMBER` rule (line ~285).

This creates a `ConstPat(IntConst(-N))` which the existing matchPattern and
MatchCompile/Exhaustive infrastructure handles automatically since IntConst
already supports any integer value.

**No changes needed to:**
- Ast.fs (IntConst already supports negative values)
- Eval.fs matchPattern (IntConst matching already works)
- MatchCompile.fs (uses `#int_N` prefix, N can be negative)
- Exhaustive.fs (ConstPat handling is generic)
- Format.fs (IntConst formatting already handles negatives)

Build and test:
```
match (0 - 1) with | -1 -> "neg one" | 0 -> "zero" | _ -> "other"
```
Expected: "neg one"

Watch for LALR(1) conflicts — MINUS in pattern context vs expression context.
If conflict, may need `%prec` annotation.
</task>

<task id="2" title="Add tuple parameter lambda to parser">
**Parser.fsy:** Add a new Expr rule for `fun (x, y) -> body`:

```
| FUN TuplePattern ARROW Expr  { ... desugar to LetPat ... }
```

The TuplePattern nonterminal already exists (line 259):
```
TuplePattern:
    | LPAREN PatternList RPAREN
```

Desugar `fun (x, y) -> body` to:
```fsharp
Lambda("__tuple_arg", LetPat(TuplePat([VarPat "x"; VarPat "y"]), Var "__tuple_arg", body))
```

This creates a lambda that takes a single tuple argument, then immediately
destructures it with a let-pattern binding.

The existing `FUN IDENT ARROW Expr` and `FUN LPAREN IDENT COLON TypeExpr RPAREN ARROW Expr`
rules handle the normal cases. The new rule matches `FUN LPAREN pattern-list RPAREN ARROW`.

**Potential LALR(1) conflict:** `FUN LPAREN` could be either:
- `fun (x, y) -> ...` (tuple pattern) — followed by PatternList RPAREN ARROW
- `fun (x : int) -> ...` (annotated param) — followed by IDENT COLON

These should disambiguate because:
- Tuple: `LPAREN IDENT COMMA ...`
- Annotated: `LPAREN IDENT COLON ...`

If there's a conflict, the annotated form's COLON vs tuple's COMMA should resolve it.

Build and test:
```
(fun (x, y) -> x + y) (1, 2)
```
Expected: 3
</task>
