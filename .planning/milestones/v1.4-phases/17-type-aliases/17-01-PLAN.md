<objective>
Implement type aliases: `type Name = string` and `type IntPair = int * int`.

Type aliases are transparent — they expand to their underlying type during elaboration.
No new runtime behavior needed; this is purely a parser + type checker feature.
</objective>

<tasks>

<task id="1" title="Add TypeAliasDecl to AST and parser">
**Ast.fs:** Add a new Decl variant:
```fsharp
| TypeAliasDecl of name: string * typeParams: string list * body: TypeExpr * Span
```

Or alternatively, reuse `TypeDecl` with a special case. Check if the parser can distinguish
`type Name = string` (alias) from `type Color = Red | Green` (ADT) at parse time.

**Key parsing challenge:** In LALR(1), `type Name = EXPR` could be either:
- Type alias: `type Name = string` (RHS is a TypeExpr)
- ADT with single constructor: `type Name = Ctor` (RHS starts with uppercase identifier)

**Strategy:** Parse `type Name = TypeExpr` as type alias when the RHS is a recognizable type expression
(int, bool, string, Type * Type, Type -> Type, Type list, etc.). If the RHS starts with an uppercase
identifier followed by `of` or `|`, it's an ADT.

**Simplest approach:** Add `TypeAliasDecl` as a new grammar rule in Parser.fsy:
```
| TYPE IDENT EQUALS TypeExpr NEWLINE  { TypeAliasDecl($2, [], $4, ruleSpan parseState 1 5) }
| TYPE IDENT TypeParams EQUALS TypeExpr NEWLINE  { TypeAliasDecl($2, $3, $5, ruleSpan parseState 1 6) }
```

Place these BEFORE the ADT `TypeDecl` rules so they take priority when the RHS is a type expression.
If LALR(1) conflicts arise, consider alternative approaches.

**Format.fs:** Add formatting for TypeAliasDecl in `formatDecl`.
</task>

<task id="2" title="Implement type alias elaboration">
**Elaborate.fs:** Add elaboration for TypeAliasDecl. The alias should be stored in a type alias
environment (Map<string, TypeExpr>) and expanded whenever the alias name is encountered as a type.

**TypeCheck.fs:** In `typeCheckDecls`, handle `TypeAliasDecl`:
1. Elaborate the TypeExpr body to a Type
2. Store in an alias environment: `Map<string, Type>`
3. When resolving type names, check alias env first and expand

**Alternative simpler approach:** Instead of a separate alias env, register the alias in the existing
typeEnv as a Scheme. For example, `type Name = string` becomes `Name -> Scheme([], TString)`.
Then type resolution naturally finds it.

**For parametric aliases** like `type Pair 'a = 'a * 'a`:
Register as `Pair -> Scheme([0], TTuple [TVar 0; TVar 0])`.

**Key:** Type aliases are fully transparent — they don't create new named types, they just expand.
`type Name = string` means `Name` IS `string`, not a distinct type.
</task>
