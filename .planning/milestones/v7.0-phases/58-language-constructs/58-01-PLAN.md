  <action>
**String slicing rules:** Find the existing rule `Atom DOTLBRACKET Expr RBRACKET` in Parser.fsy (produces IndexGet). Add TWO new rules immediately after it:

```fsharp
| Atom DOTLBRACKET Expr DOTDOT Expr RBRACKET
    { StringSliceExpr($1, $3, Some $5, ruleSpan parseState 1 6) }
| Atom DOTLBRACKET Expr DOTDOT RBRACKET
    { StringSliceExpr($1, $3, None, ruleSpan parseState 1 5) }
```

LALR note: The parser sees `Atom DOTLBRACKET Expr` then peeks lookahead — `RBRACKET` reduces as IndexGet, `DOTDOT` shifts to the new slice rules. No conflict.

**List comprehension rules:** Find existing list-related Atom rules (e.g., `LBRACKET Expr DOTDOT Expr RBRACKET` for Range, or `LBRACKET RBRACKET` for empty list). Add TWO new rules near the other LBRACKET rules:

```fsharp
| LBRACKET FOR IDENT IN Expr ARROW Expr RBRACKET
    { ListCompExpr($3, $5, $7, ruleSpan parseState 1 8) }
| LBRACKET FOR IDENT IN Expr DOTDOT Expr ARROW Expr RBRACKET
    { ListCompExpr($3, Range($5, $7, None, ruleSpan parseState 5 7), $9, ruleSpan parseState 1 10) }
```

The second rule desugars `[for i in 0..4 -> body]` into `ListCompExpr(i, Range(0,4,None), body)` — Range already evaluates to ListValue so no extra machinery needed.

After adding rules, rebuild to check for LALR conflicts (fsyacc will report them during build).
