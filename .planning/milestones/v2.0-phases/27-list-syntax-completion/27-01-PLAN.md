<objective>
Add bracket depth tracking to IndentFilter so that INDENT/DEDENT tokens are suppressed inside [], (), and {} brackets. This makes multi-line list literals (SYN-02) parse correctly.

Purpose: The IndentFilter currently emits INDENT/DEDENT for any NEWLINE, even inside bracketed expressions. This causes parse failures for multi-line list literals.
Output: Updated IndentFilter.fs with BracketDepth in FilterState, updated test fixtures, and new flt tests confirming multi-line lists work.
</objective>

  <name>Task 1: Add BracketDepth to FilterState and suppress INDENT/DEDENT inside brackets</name>
  <files>src/LangThree/IndentFilter.fs</files>
  <action>
    1. Add `BracketDepth: int` field to the `FilterState` record (after `PrevToken`).

    2. Update `initialState` to include `BracketDepth = 0`.

    3. In the `filter` function's main `match token with` expression, add handling for bracket open/close tokens BEFORE the existing NEWLINE handler:
       - `Parser.LBRACKET | Parser.LPAREN | Parser.LBRACE` -> increment BracketDepth by 1, yield token
       - `Parser.RBRACKET | Parser.RPAREN | Parser.RBRACE` -> decrement using `max 0 (state.BracketDepth - 1)`, yield token

    4. Add a new NEWLINE guard arm BEFORE the existing `| Parser.NEWLINE col ->` arm:
       ```fsharp
       | Parser.NEWLINE _ when state.BracketDepth > 0 ->
           state <- { state with LineNum = state.LineNum + 1 }
           // yield nothing — suppress INDENT/DEDENT inside brackets
       ```
       CRITICAL: This guarded arm MUST appear before the non-guarded `| Parser.NEWLINE col ->` arm. F# match arms are top-to-bottom; if the non-guarded arm comes first, the bracket depth check never fires.

    5. If bracket tokens were previously handled in a catch-all or other arm, remove that duplication and replace with the new explicit arms.
  </action>
  <verify>dotnet build src/LangThree/LangThree.fsproj</verify>
  <done>Build succeeds with no errors. The BracketDepth field exists in FilterState and initialState.</done>
</task>
