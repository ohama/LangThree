<objective>
Fix IndentFilter so that `else` followed by indented expression keywords parses correctly.

Purpose: When `else` is on a line that is more indented than the `if`, IndentFilter currently emits INDENT before ELSE, which confuses the parser. The fix suppresses this INDENT while keeping all DEDENT tokens (needed to close the THEN branch).

Output: Modified IndentFilter.fs + new IndentFilter tests for the ELSE case.
</objective>

In `processNewlineWithContext` (around line 193), the final `| _ ->` branch returns `(stateAfterIndent, indentTokens)`. Modify this branch to suppress any INDENT token when `nextToken = Some Parser.ELSE`:

```fsharp
| _ ->
    // Suppress INDENT when next token is ELSE
    // (ELSE follows THEN directly — INDENT between them confuses the parser)
    // DEDENTs are NOT suppressed — they close open blocks from the THEN branch.
    let filteredTokens =
        match nextToken with
        | Some Parser.ELSE when List.contains Parser.INDENT indentTokens ->
            indentTokens |> List.filter (fun t -> t <> Parser.INDENT)
        | _ -> indentTokens
    (stateAfterIndent, filteredTokens)
```

Key points:
- `indentTokens` is the result from the `processNewline` call in the preceding lines (stored in `tokens` or `indentTokens` — check the exact binding name at line 193-195 in the file).
- The indent stack IS updated normally (via `processNewline`) — only the emitted INDENT token is suppressed.
- DEDENT tokens pass through unchanged.
- This fix is ONLY needed in the `| _ ->` branch. The `InMatch` and `InTry` branches (lines 180-192) already return `[]` so they are unaffected.

After the fix, verify by looking at the `| _ ->` branch and confirming `nextToken` is in scope (it is — it's a parameter of `processNewlineWithContext` at line 111).
