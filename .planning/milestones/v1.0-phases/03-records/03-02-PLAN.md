5. CRITICAL -- LBRACE IDENT AMBIGUITY RESOLUTION:

   The parser faces an LALR(1) ambiguity with `LBRACE IDENT`:
   - Record creation: `{ x = 1; y = 2 }` -- IDENT is a field name, next token is EQUALS
   - Copy-and-update: `{ expr with y = 3 }` -- IDENT could be start of Expr

   fsyacc is LALR(1) and CANNOT backtrack. The solution is to use TWO separate Atom productions instead of a shared RecordExprInner:

   ```
   // Record creation: LBRACE followed by field bindings
   | LBRACE RecordFieldBindings RBRACE { RecordExpr(None, $2, ruleSpan parseState 1 3) }
   // Copy-and-update: LBRACE followed by an expression, then WITH
   | LBRACE Expr WITH RecordFieldBindings RBRACE { RecordUpdate($2, $4, ruleSpan parseState 1 5) }
   ```

   WHY THIS WORKS without conflict:
   - RecordFieldBindings starts with `IDENT EQUALS` (two tokens of lookahead resolved by grammar structure)
   - Expr can start with IDENT, but after IDENT the parser sees WITH (not EQUALS)
   - The key discrimination: after `LBRACE IDENT`, the NEXT token is either:
     - EQUALS -> parser is in RecordFieldBindings (record creation)
     - WITH -> parser reduced IDENT to Expr, now in copy-and-update
     - Other (DOT, LPAREN, etc.) -> parser continues reducing Expr, eventually sees WITH

   In LALR(1) terms: after shifting IDENT, the parser has it on the stack. The lookahead token determines the action:
   - Lookahead EQUALS: shift EQUALS as part of `IDENT EQUALS Expr` in RecordFieldBindings
   - Lookahead WITH: reduce IDENT to Expr (via Atom -> Expr chain), then shift WITH

   This is conflict-free because EQUALS and WITH are distinct lookahead tokens that unambiguously select the production.

   IMPORTANT: If fsyacc still reports a shift/reduce conflict on `LBRACE IDENT`, it may be because the grammar reduces IDENT to Atom to Expr BEFORE seeing the next token. In that case, use the RecordExprInner approach with a function return instead:
   ```
   | LBRACE RecordExprInner RBRACE  { $2 (ruleSpan parseState 1 3) }

   RecordExprInner:
       | RecordFieldBindings
           { fun span -> RecordExpr(None, $1, span) }
       | Expr WITH RecordFieldBindings
           { fun span -> RecordUpdate($1, $3, span) }
   ```
   With RecordExprInner, the parser speculatively tries RecordFieldBindings first (IDENT EQUALS is unambiguous), and if it doesn't match (no EQUALS after IDENT), falls through to Expr WITH. This works because RecordFieldBindings requires IDENT EQUALS, which is NOT a valid Expr continuation.

   AFTER BUILDING: Check `dotnet build` output for parser conflicts. If you see shift/reduce or reduce/reduce warnings mentioning LBRACE, IDENT, EQUALS, or WITH, switch to whichever approach avoids the conflict. Document which approach was used in the SUMMARY.
