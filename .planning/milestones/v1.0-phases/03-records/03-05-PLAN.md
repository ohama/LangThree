Create `RecordTests.fs` with tests covering all record requirements. The tests should use the same patterns as existing tests (likely parsing source code strings and running through the full pipeline).

Test categories:

**REC-01: Record type declarations**
- Simple record: `type Point = { x: int; y: int }`
- Record with type parameters: `type Pair 'a = { first: 'a; second: 'a }`
- Trailing semicolon: `type R = { x: int; }`

**REC-02: Record expressions**
- Create record: `type Point = { x: int; y: int }\nlet p = { x = 1; y = 2 }`
- Verify field values via access

**REC-03: Field access**
- Simple access: `p.x` returns field value
- Chained access: If a record has a record field, `p.inner.x` works
- Type checking: accessing non-existent field produces error

**REC-04: Copy-and-update**
- `{ p with y = 3 }` creates new record with y changed, x preserved
- Update multiple fields: `{ p with x = 10; y = 20 }`
- Original record unchanged after update

**REC-05: Record patterns**
- Match on record: `match p with { x = px; y = py } -> px + py`
- Partial match: `match p with { x = px } -> px` (only bind x)

**REC-06: Structural equality**
- Two records with same fields are equal
- Two records with different field values are not equal

**Error cases:**
- Duplicate field names across types: error
- Wrong field names in record expression: error
- Missing fields in record expression: error
- Field access on non-record: error

Each test should be a full program string parsed and evaluated through the module pipeline:
1. Parse with `parseModule`
2. Type check with `typeCheckModule`
3. Evaluate (if type check passes)

If the existing test framework does not support module-level testing, check how Phase 2 tests work and follow the same pattern.

This task handles fixes discovered during Task 1 testing. Common issues to anticipate:

1. **RecordExpr type name resolution:** The evaluator creates RecordValue with empty type name. If structural equality depends on type name matching, this needs fixing. Options:
   - Pass resolved type name through a separate mechanism
   - Look up type name from field names in eval (requires RecordEnv access)
   - Skip type name comparison in equality (compare field maps only)
   The simplest fix: in RecordExpr evaluation, if the type name matters, you can encode it in the AST during a type-checking pass. But since our evaluator runs post-type-check, and equality uses `t1 = t2`, mismatched empty strings would incorrectly make different record types equal. Fix: resolve type name in RecordExpr eval by matching field names against known record types. OR change the equality check to only compare Maps.

   Recommended: For structural equality (REC-06), specify that records of the SAME type with equal fields are equal. Since the type checker ensures correct typing, the type name in RecordValue is informational. Set it to empty string and compare only field maps in Equal/NotEqual. This matches the semantic that type checking already ensures type safety.

2. **RecordPat in type checker:** The pattern in Infer.fs returns a fresh type var. The Match handler in Bidir.fs needs to unify this with the scrutinee type. Verify this works through testing.

3. **Missing field validation in RecordExpr:** The type checker should validate that ALL required fields are present and NO extra fields exist. Test this explicitly.

4. **Parser edge cases:** Multi-line records with indentation, trailing semicolons, records inside function bodies.

Fix any issues found. If no issues are found, this task completes immediately.
