   ```
   | TEData of name: string * args: TypeExpr list   // T<'a, 'b> or int expr

   type ConstructorInfo = {

       ArgType: Type option
       ResultType: Type
       IsGadt: bool
       ExistentialVars: int list
   }
   ```
   IsGadt defaults to false for regular ADT constructors.
   ExistentialVars lists type parameter indices that appear in arg types but NOT in the return type (empty for non-GADT).

   IMPORTANT: Do NOT add LocalConstraints to ConstructorInfo. Local constraints are computed at match-time by unifying the constructor's return type with the scrutinee type. They are dynamic, not static metadata.

<task type="auto">
  <name>Task 2: Parser grammar for GADT constructor syntax</name>
  <files>src/LangThree/Parser.fsy, src/LangThree/Elaborate.fs</files>
  <action>
1. In Parser.fsy, extend the `Constructor` rule to support GADT syntax. GADT constructors use OCaml-style colon syntax within type declarations:
   ```
   Constructor:
       | IDENT                             { Ast.ConstructorDecl($1, None, symSpan parseState 1) }
       | IDENT OF TypeExpr                 { Ast.ConstructorDecl($1, Some $3, ruleSpan parseState 1 3) }
       // GADT: nullary with explicit return type (e.g., Unit : unit expr)
       | IDENT COLON TypeExpr              { Ast.GadtConstructorDecl($1, [], $3, ruleSpan parseState 1 3) }
       // GADT: with arguments (e.g., Int : int -> int expr)
       | IDENT COLON GadtArgTypes ARROW TypeExpr
           { Ast.GadtConstructorDecl($1, $3, $5, ruleSpan parseState 1 5) }
   ```

   Add a new rule `GadtArgTypes` that parses the argument types before the arrow. The arguments can be:
   - Single type: `int` in `Int : int -> int expr`
   - Tuple of types: `int expr * int expr` in `Add : int expr * int expr -> int expr`

   ```
   GadtArgTypes:
       | GadtArgType STAR GadtArgTypes   { $1 :: $3 }
       | GadtArgType                     { [$1] }

   GadtArgType:
       | AtomicType                      { $1 }
       | AtomicType IDENT               { TEData($2, [$1]) }
       | LPAREN TypeExpr RPAREN          { $2 }
   ```

   Wait -- this approach has LALR conflicts. Let me reconsider.

   Actually, the simpler approach: reuse the existing TypeExpr grammar for GADT arg types. The `IDENT COLON TypeExpr ARROW TypeExpr` rule works because:
   - `Int : int -> int expr` -- TypeExpr before ARROW is `int`, TypeExpr after is `int expr`
   - `Add : int expr * int expr -> int expr` -- TypeExpr before ARROW is `int expr * int expr` (parsed as TupleType)

   But we need TypeExpr to support applied named types like `int expr`. Currently `AtomicType` has `IDENT` -> `TEName` but not `AtomicType IDENT` for type application.

   Add a type application rule in AtomicType:
   ```
   | AtomicType IDENT   { TEData($2, [$1]) }
   ```
   This handles `int expr`, `bool expr`, `'a expr` etc. (postfix type application like F#/OCaml).

   Then the GADT constructor rules become simply:
   ```
   | IDENT COLON TypeExpr              { Ast.GadtConstructorDecl($1, [], $3, ruleSpan parseState 1 3) }
   | IDENT COLON TypeExpr ARROW TypeExpr
       { let args = match $3 with TETuple ts -> ts | t -> [t]
         Ast.GadtConstructorDecl($1, args, $5, ruleSpan parseState 1 5) }
   ```

   For the nullary case `IDENT COLON TypeExpr`, this represents a GADT constructor with no arguments that just returns a specific type. However, be careful: this conflicts with `IDENT COLON TypeExpr ARROW TypeExpr` since `TypeExpr` can contain arrows. The parser will handle this correctly because `IDENT COLON TypeExpr` matches only when the next token is NOT ARROW (i.e., it's PIPE, DEDENT, or EOF).

   Actually, reconsider: `IDENT COLON TypeExpr` where TypeExpr could be an arrow type is ambiguous with `IDENT COLON TypeExpr ARROW TypeExpr`. The LALR parser will have a shift-reduce conflict on ARROW. We should NOT have the nullary form as a separate rule. Instead:
   - `IDENT COLON TypeExpr` -- This IS the GADT constructor. If TypeExpr has no arrow, it's nullary (just return type). If TypeExpr has arrow(s), the last arrow separates args from return type.

   Simplest approach: just use ONE rule:
   ```
   | IDENT COLON TypeExpr
       { // Parse the full type expression, then split into args and return type
         // If it's an arrow type A -> B, args = [A], return = B (or split tuple A)
         // If it's not an arrow type, it's a nullary constructor with that return type
         let rec splitGadtType (te: TypeExpr) =
             match te with
             | TEArrow(argTy, retTy) ->
                 // Check if retTy is also an arrow (curried args)
                 let moreArgs, finalRet = splitGadtType retTy
                 match moreArgs with
                 | [] ->
                     // retTy is the final return type
                     let args = match argTy with TETuple ts -> ts | t -> [t]
                     (args, retTy)
                 | _ ->
                     // retTy had more arrows: prepend this arg
                     (argTy :: moreArgs, finalRet) // Hmm, this doesn't work for tupled args
             | other -> ([], other)
         let args, retTy = splitGadtType $3
         Ast.GadtConstructorDecl($1, args, retTy, ruleSpan parseState 1 3) }
   ```

   Wait, this is getting complex in the parser action. Let's keep it simple. Use TWO rules but avoid the conflict:

   Better approach -- just one rule `IDENT COLON TypeExpr` and do the splitting in F# code:
   ```
   | IDENT COLON TypeExpr
       { let rec splitGadt (te: Ast.TypeExpr) =
             match te with
             | Ast.TEArrow(args, ret) ->
                 let argList = match args with Ast.TETuple ts -> ts | t -> [t]
                 (argList, ret)
             | t -> ([], t)  // Nullary GADT constructor
         let (args, ret) = splitGadt $3
         Ast.GadtConstructorDecl($1, args, ret, ruleSpan parseState 1 3) }
   ```

   This works because the arrow type parser is right-associative, so `int * int -> int -> int expr` parses as `TEArrow(TETuple[int,int], TEArrow(int, int expr))`. The splitGadt only peels off the outermost arrow, giving args=[int,int] and ret=TEArrow(int, int expr)... which is wrong, we want the LAST arrow to separate args from return type.

   Actually for GADTs: `Add : int expr * int expr -> int expr` has exactly ONE arrow. Multi-argument constructors use tuples, not currying. So single-arrow split is correct. For curried GADT constructors we would need multiple arrows but that's not standard -- GADT constructors take a single (possibly tupled) argument.

   So the simple splitGadt above is correct: peel the first (and only) arrow. If there's no arrow, it's nullary.

   This is the approach to implement.
