# Milestone v1.0: Core Language

**Status:** SHIPPED 2026-03-10
**Phases:** 1-7
**Total Plans:** 32

## Overview

LangThree transforms FunLang v6.0 into a practical ML-style functional language with F# syntax and modern type system features. The roadmap progresses through seven phases: first establishing indentation-based parsing (foundation for all syntax), then building out the type system with ADTs and GADTs (core expressiveness), adding records (practical data structures), implementing modules (code organization), exceptions (error handling), and finally optimized pattern matching compilation to decision trees.

## Phases

### Phase 1: Indentation-Based Syntax
**Goal**: Parser accepts F# style indentation-based syntax for all language constructs
**Depends on**: Nothing (first phase)
**Requirements**: INDENT-01 through INDENT-08
**Plans**: 4 plans

Plans:
- [x] 01-01: Add context-aware indentation processing for match expressions with pipe alignment
- [x] 01-02: Enable multi-line function application with indented argument grouping
- [x] 01-03: Improve indentation error messages and add configurable indent width validation
- [x] 01-04: Support module-level declarations with multiple top-level let bindings

## Milestone Summary

**Key Decisions:**
- F# style syntax over OCaml (indentation-based is more modern, simpler)
- GADT included (leverages FunLang's bidirectional type checking)
- Functor excluded (complexity vs. practical value too low)
- Jules Jacobs algorithm for pattern matching (binary trees, simpler than Maranget N-way)
- evalFn parameter pattern to avoid circular F# module dependencies

**Issues Resolved:**
- TExn unification case missing in Unify.fs (Phase 6)
- InTry context popping uses strict < (Phase 6)
- Record constructor encoding changed to field-name-based for partial patterns (Phase 7)
- 4 bugs found and fixed during exception integration testing

**Technical Debt Incurred:**
- Nested indentation-based let requires explicit `in` keywords (workaround exists)
- Decision tree compiled on every match evaluation (no caching)
- Two-phase compilation design needed for modules (deferred)
