```fsharp
/// Desugar mixed parameter list (plain + annotated) into nested Lambda/LambdaAnnot chain
let rec desugarMixedParams (paramList: Choice<string, string * TypeExpr> list) (body: Expr) (span: Span) : Expr =
    match paramList with
    | [] -> body
    | Choice1Of2 name :: rest ->
        Lambda(name, desugarMixedParams rest body span, span)
    | Choice2Of2 (name, ty) :: rest ->
        LambdaAnnot(name, ty, desugarMixedParams rest body span, span)
```

2. Add new grammar rules `MixedParam` and `MixedParamList` AFTER the existing `ParamList` rule (after line 700):

```
MixedParamList:
    | MixedParam                   { [$1] }
    | MixedParam MixedParamList    { $1 :: $2 }

MixedParam:
    | IDENT                                    { Choice1Of2 $1 }
    | LPAREN IDENT COLON TypeExpr RPAREN       { Choice2Of2 ($2, $4) }
```

The nonterminal `MixedParamList` must be declared with `%type` in the declarations section. Add:
```

```
Place these near the existing `%type` declarations (look for `%type <(string * TypeExpr) list> AnnotParamList` and add nearby).

**IMPORTANT ORDERING:** In each grammar section, place the MixedParamList productions AFTER the existing ParamList productions. FsLexYacc tries alternatives top-to-bottom, so ParamList (plain params only) should be tried first to avoid ambiguity — MixedParamList is only needed when annotated params appear. Actually, since MixedParamList's first token can be IDENT (same as ParamList), there WILL be an LALR ambiguity. To resolve this: place MixedParamList productions BEFORE existing ParamList productions (so the parser prefers MixedParamList when it sees IDENT, which is fine because MixedParam includes plain IDENT via Choice1Of2). This way MixedParamList subsumes ParamList. Alternatively, keep BOTH — FsLexYacc will pick the first matching rule. If build shows shift/reduce conflicts, remove the original ParamList-based let productions and rely solely on MixedParamList (which handles all-plain params as a degenerate case via Choice1Of2).

Watch for conflicts: If `dotnet build` shows parser conflicts, the resolution is to REMOVE the duplicate ParamList productions (since MixedParamList subsumes them). MixedParamList with all Choice1Of2 items produces the same Lambda chain as the old ParamList foldBack.
