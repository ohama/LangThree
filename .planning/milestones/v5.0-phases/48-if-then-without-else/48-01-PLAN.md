<objective>
Add `if cond then expr` (without else) as syntactic sugar for `if cond then expr else ()`.

Purpose: Enables idiomatic imperative-style code like `if x > 0 then print "positive"` without requiring an explicit else branch. The desugaring approach means zero changes to AST, Eval, Bidir, Infer, TypeCheck, or Format — the parser alone handles this by synthesizing a unit else-branch.

Output: One new grammar rule in Parser.fsy, regenerated Parser.fs, and 3 flt integration tests covering IFTHEN-01 and IFTHEN-02.
</objective>

  <name>Task 1: Add if-then-without-else grammar rule to Parser.fsy and regenerate Parser.fs</name>
  <files>src/LangThree/Parser.fsy, src/LangThree/Parser.fs</files>
  <action>
In `src/LangThree/Parser.fsy`, locate the existing if-then-else rule (line ~151):

```
| IF Expr THEN SeqExpr ELSE SeqExpr    { If($2, $4, $6, ruleSpan parseState 1 6) }
```

Add a new rule IMMEDIATELY AFTER it (keep the else variant first so the parser prefers it):

```
| IF Expr THEN SeqExpr                 { If($2, $4, Tuple([], symSpan parseState 4), ruleSpan parseState 1 4) }
```

This desugars `if cond then expr` to `If(cond, expr, Tuple([], unit_span), full_span)`, exactly like `if cond then expr else ()`. The `symSpan parseState 4` references the THEN-branch token position (token 4 = SeqExpr) to attach the synthetic unit span to the end of the then-branch. `ruleSpan parseState 1 4` covers the full rule from IF (token 1) to SeqExpr (token 4).

Do NOT add the rule before the else variant — LALR(1) shift/reduce resolution will handle the ambiguity correctly when ELSE appears (it shifts), but keeping the longer rule first is cleaner intent.

After editing Parser.fsy, regenerate Parser.fs using the project's build-time code generation:

```bash
dotnet build src/LangThree/LangThree.fsproj -c Release
```

The build invokes fsyacc to regenerate Parser.fs from Parser.fsy. If the build succeeds, Parser.fs is correctly regenerated. If there is a shift/reduce conflict warning for the dangling-else problem, that is expected and acceptable — the parser resolves it by shifting (preferring the else branch when ELSE appears), which is the correct behavior.

Verify the new rule appears in Parser.fs:
```bash
grep "Tuple(\[\]" src/LangThree/Parser.fs
```
