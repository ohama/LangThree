  <files>src/LangThree/Parser.fsy</files>
  <action>
1. Add token declarations at the top of Parser.fsy (after existing token declarations):
   ```
   %token LBRACE RBRACE SEMICOLON DOT
   ```
   Do NOT add MUTABLE or LARROW yet (Plan 06).

2. Add RecordDeclaration grammar rule in the Decls section. Add it alongside TypeDeclaration:
   ```
   // In Decls rule, add:
   | RecordDeclaration               { [Ast.Decl.RecordTypeDecl $1] }
   | RecordDeclaration Decls         { (Ast.Decl.RecordTypeDecl $1) :: $2 }
   ```

3. Add RecordDeclaration grammar rule:
   ```
   RecordDeclaration:
       | TYPE IDENT TypeParams EQUALS LBRACE RecordFields RBRACE
           { Ast.RecordDecl($2, $3, $6, ruleSpan parseState 1 7) }
       | TYPE IDENT TypeParams EQUALS INDENT LBRACE RecordFields RBRACE DEDENT
           { Ast.RecordDecl($2, $3, $7, ruleSpan parseState 1 9) }
   ```

4. Add RecordFields and RecordField rules:
   ```
   RecordFields:
       | RecordField                              { [$1] }
       | RecordField SEMICOLON RecordFields       { $1 :: $3 }
       | RecordField SEMICOLON                    { [$1] }

   RecordField:
       | IDENT COLON TypeExpr
           { Ast.RecordFieldDecl($1, $3, false, ruleSpan parseState 1 3) }
   ```
   Note: MUTABLE field syntax is deferred to Plan 06.

5. Add record expressions in Atom rule (field access and record literal/update):
   ```
   // In Atom, add:
   // Field access (left-recursive for chaining: a.b.c)
   | Atom DOT IDENT    { FieldAccess($1, $3, ruleSpan parseState 1 3) }
   // Record expression and copy-and-update
   | LBRACE RecordExprInner RBRACE  { $2 (ruleSpan parseState 1 3) }
   ```

6. Add RecordExprInner rule to distinguish creation vs copy-and-update:
   ```
   RecordExprInner:
       | RecordFieldBindings
           { fun span -> RecordExpr(None, $1, span) }
       | Expr WITH RecordFieldBindings
           { fun span -> RecordUpdate($1, $3, span) }
   ```

7. Add RecordFieldBindings rule:
   ```
   RecordFieldBindings:
       | IDENT EQUALS Expr                              { [($1, $3)] }
       | IDENT EQUALS Expr SEMICOLON RecordFieldBindings { ($1, $3) :: $5 }
       | IDENT EQUALS Expr SEMICOLON                    { [($1, $3)] }
   ```

8. Add RecordPat in Pattern rule:
   ```
   // In Pattern, add:
   | LBRACE RecordPatFields RBRACE  { RecordPat($2, ruleSpan parseState 1 3) }
   ```

9. Add RecordPatFields rule:
   ```
   RecordPatFields:
       | IDENT EQUALS Pattern                              { [($1, $3)] }
       | IDENT EQUALS Pattern SEMICOLON RecordPatFields    { ($1, $3) :: $5 }
       | IDENT EQUALS Pattern SEMICOLON                    { [($1, $3)] }
   ```

CRITICAL PARSER NOTES:
- The `Atom DOT IDENT` rule creates left-recursion in Atom, which is fine for LR parsers (it enables chaining `a.b.c`).
- `EQUALS` is already a token used for both `=` comparison and `let x = ...`. In record field bindings `{ x = 1 }`, the grammar context is unambiguous because it appears inside braces after an IDENT.
- The `RecordExprInner` rule uses a function `fun span -> ...` to thread the brace span. This is a common fsyacc pattern.
- If the function-returning approach causes issues with fsyacc, use a simpler approach: split into two Atom productions:
  ```
  | LBRACE RecordFieldBindings RBRACE { RecordExpr(None, $2, ruleSpan parseState 1 3) }
  | LBRACE Expr WITH RecordFieldBindings RBRACE { RecordUpdate($2, $4, ruleSpan parseState 1 5) }
  ```
  This may cause shift/reduce conflicts. If so, the `RecordExprInner` approach with the function return is the safer option.
