  <name>Task 1: Add AST-to-CasePat conversion, constructor set lookup, and warning types</name>
  <files>
    src/LangThree/Exhaustive.fs

  </files>
  <action>
**In Exhaustive.fs:**

1. Add `open Type` at the top (needed for ConstructorEnv/ConstructorInfo access).

2. Add a function `getConstructorsFromEnv` that takes a `Type.ConstructorEnv` and a scrutinee `Type.Type`, and returns a `ConstructorSet` (the Exhaustive module's own ConstructorInfo list). The logic:
   - If scrutinee type is `TData(typeName, _)`, filter the ConstructorEnv for all entries whose `ResultType` is `TData(typeName, _)` (same type name). For each matching entry, create an `Exhaustive.ConstructorInfo` with `Name = constructorName` and `Arity = if argType.IsSome then 1 else 0`. Note: multi-field constructors use tuple args, so arity is always 0 or 1.
   - If scrutinee type is not TData, return an empty list (non-ADT types have no constructor set for exhaustiveness).

3. Replace the `getConstructors` failwith stub (lines 227-228) with a call through to `getConstructorsFromEnv` or just remove it entirely since we will pass constructor sets explicitly from TypeCheck.

4. Add a function `astPatToCasePat` that converts `Ast.Pattern` to `Exhaustive.CasePat`:
   - `Ast.ConstructorPat(name, Some argPat, _)` -> `Exhaustive.ConstructorPat(name, [astPatToCasePat argPat])`
   - `Ast.ConstructorPat(name, None, _)` -> `Exhaustive.ConstructorPat(name, [])`
   - `Ast.WildcardPat _` -> `Exhaustive.WildcardPat`
   - `Ast.VarPat _` -> `Exhaustive.WildcardPat` (variable patterns match everything)
   - `Ast.TuplePat _` -> `Exhaustive.WildcardPat` (not relevant for ADT exhaustiveness)
   - `Ast.ConsPat _` -> `Exhaustive.WildcardPat` (list patterns - not ADT)
   - `Ast.EmptyListPat _` -> `Exhaustive.WildcardPat` (list patterns - not ADT)
   - `Ast.ConstPat _` -> `Exhaustive.WildcardPat` (constant patterns - not ADT)

   This function needs `open Ast` but there is a name collision: both Ast and Exhaustive define `ConstructorPat` and `WildcardPat`. Use fully qualified names: `Ast.ConstructorPat`, `Ast.WildcardPat`, etc. in the match, and `Exhaustive.ConstructorPat`, `Exhaustive.WildcardPat` in the return values. Since the module is `Exhaustive`, return values can use unqualified names (they resolve to Exhaustive types). The match patterns need qualification since `open Ast` would shadow.

   Alternative approach (preferred): Do NOT open Ast. Instead, qualify all Ast references as `Ast.Pattern`, `Ast.ConstructorPat`, etc. The function signature: `let rec astPatToCasePat (pat: Ast.Pattern) : CasePat`.

**In Diagnostic.fs:**

5. Add two new `TypeErrorKind` variants for warnings:
   - `NonExhaustiveMatch of missingPatterns: string list` -- the string list contains formatted missing pattern witnesses
   - `RedundantPattern of index: int` -- the 0-based index of the redundant clause

6. Add corresponding cases in `typeErrorToDiagnostic`:
   - `NonExhaustiveMatch patterns` -> code "W0001", message "Incomplete pattern match. Missing cases: {patterns joined with ', '}", hint "Add the missing cases or a wildcard pattern '_' to cover all values"
   - `RedundantPattern idx` -> code "W0002", message "Redundant pattern in clause {idx + 1}. This case will never be reached.", hint "Remove the unreachable pattern"

   Note: Using "W" prefix for warning codes to distinguish from "E" error codes.

7. In `formatDiagnostic`, change the header line to use "warning" instead of "error" when the code starts with "W":
   - If `code` starts with "W": `sb.AppendLine(sprintf "warning[%s]: %s" code diag.Message)`
   - Otherwise keep existing "error[%s]" format

<task type="auto">
  <name>Task 2: Wire exhaustiveness and redundancy checking into TypeCheck.fs with integration tests</name>

**In TypeCheck.fs:**

1. Add `open Exhaustive` at the top (Exhaustive.fs is compiled before TypeCheck.fs per fsproj order).

2. Change `typeCheckModule` return type from `Result<unit, Diagnostic>` to `Result<Diagnostic list, Diagnostic>`. The Ok case now carries a list of warning diagnostics (may be empty). The Error case still carries a single fatal error diagnostic. Update every existing `Ok ()` return site to `Ok []` (empty warnings list).

3. After the existing let-declaration type checking fold (which produces `_finalEnv`), add exhaustiveness/redundancy checking:

   a. Collect all match expressions from the module's let declarations using recursive AST traversal. Write a recursive helper `collectMatches (expr: Ast.Expr) : (Ast.Pattern list * Ast.Expr * Ast.Span) list` that traverses the AST and collects `(patterns, scrutinee, span)` for each `Match(scrutinee, clauses, span)` expression. The patterns come from `clauses |> List.map fst`. Recurse into all sub-expressions: Let body+rhs, LetRec body+rhs, If branches, Match clause bodies+scrutinee, Lambda body, App fun+arg, etc. This recursive approach is needed to catch nested match expressions (e.g., match inside a lambda inside a let).

   b. For each collected match expression, determine the scrutinee type by calling `Bidir.synth ctorEnv [] env scrutinee`. Note: this re-synthesizes from scratch (no caching), but for TData name extraction the result is consistent -- if the scrutinee was typed as `TData("Option", _)` during the first pass, re-synthesis will produce a fresh `TData("Option", _)` with the same type name. Extract the result type from the substitution. Use this to get the constructor set via `getConstructorsFromEnv ctorEnv scrutineeType`.

   c. If the constructor set is non-empty (meaning it's an ADT match):
      - Convert each `Ast.Pattern` to `CasePat` using `astPatToCasePat`
      - Call `Exhaustive.checkExhaustive constructorSet casePatterns`
      - Call `Exhaustive.checkRedundant constructorSet casePatterns`
      - For `NonExhaustive missing`: construct a full `TypeError` record and convert to Diagnostic:
        ```fsharp
        { Kind = NonExhaustiveMatch(missing |> List.map Exhaustive.formatPattern)
          Span = matchSpan
          Term = None
          ContextStack = []
          Trace = [] }
        |> typeErrorToDiagnostic
        ```
      - For `HasRedundancy indices`: create a warning Diagnostic for each redundant index using the same full `TypeError` record pattern:
        ```fsharp
        { Kind = RedundantPattern(idx)
          Span = matchSpan  // or Ast.patternSpanOf (patterns.[idx]) if available
          Term = None
          ContextStack = []
          Trace = [] }
        |> typeErrorToDiagnostic
        ```

   d. Collect all warnings into a list and return `Ok(warnings)`.

   **Important:** The exhaustiveness check should NOT cause type checking to fail. Warnings are informational. The existing error handling (TypeException) remains unchanged for real type errors.

4. Update callers of `typeCheckModule` to handle the new `Result<Diagnostic list, Diagnostic>` return type:
   - Check `Program.fs` and `Repl.fs` for calls to `typeCheckModule`.
   - On `Ok warnings`: print each warning via `formatDiagnostic` to stderr/console, then continue execution.
   - On `Error diag`: keep existing error handling unchanged.

Change it to return `Result<Diagnostic list, Diagnostic>` to match the new `typeCheckModule` signature. This is a direct passthrough -- no logic change needed.

      Assert that `typeCheckModule` returns `Ok(warnings)` where warnings is non-empty and contains a diagnostic with code `Some "W0001"` and message containing "None" (the missing case).

   b. `testExhaustivenessNoWarningComplete`: Parse and type-check a module with complete match:

      Assert that `typeCheckModule` returns `Ok(warnings)` where warnings contains a diagnostic with code `Some "W0002"`.
