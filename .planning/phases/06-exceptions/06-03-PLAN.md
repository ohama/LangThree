<task type="auto">
  <name>Task 1: Exception runtime evaluation and when guards</name>
  <files>src/LangThree/Eval.fs</files>
  <action>
1. Define the .NET exception type near the top of Eval.fs (after opens, before eval):
   ```fsharp
   exception LangThreeException of Value
   ```

2. Replace the Raise stub in `eval` with:
   ```fsharp
   | Raise(arg, _) ->
       let exnVal = eval recEnv moduleEnv env arg
       raise (LangThreeException exnVal)
   ```

3. Replace the TryWith stub in `eval` with:
   ```fsharp
   | TryWith(body, handlers, _) ->
       try
           eval recEnv moduleEnv env body
       with
       | LangThreeException exnVal ->
           evalMatchClauses recEnv moduleEnv env exnVal handlers
   ```

4. Update `evalMatchClauses` to handle when guards. Currently it destructures `(pattern, _guard, resultExpr)` but ignores the guard (from Plan 01). Change to:
   ```fsharp
   and evalMatchClauses recEnv moduleEnv env scrutinee clauses =
       match clauses with
       | [] -> failwith "Match failure: no pattern matched"
       | (pattern, guard, resultExpr) :: rest ->
           match matchPattern pattern scrutinee with
           | Some bindings ->
               let extendedEnv = List.fold (fun e (n, v) -> Map.add n v e) env bindings
               // Check when guard if present
               match guard with
               | None ->
                   eval recEnv moduleEnv extendedEnv resultExpr
               | Some guardExpr ->
                   match eval recEnv moduleEnv extendedEnv guardExpr with
                   | BoolValue true -> eval recEnv moduleEnv extendedEnv resultExpr
                   | _ -> evalMatchClauses recEnv moduleEnv env scrutinee rest
           | None ->
               evalMatchClauses recEnv moduleEnv env scrutinee rest
   ```

5. In `evalModuleDecls` (the module-level declaration evaluator), add handling for `ExceptionDecl`:
   - `ExceptionDecl(name, dataTypeOpt, _)`: Register the exception constructor in the module environment.
   - If nullary (dataTypeOpt = None): add to env as `ExnValue(name, None)`
   - If has data (dataTypeOpt = Some _): add to env as `FunctionValue` that wraps arg in `ExnValue(name, Some arg)`. Use a lambda like: `FunctionValue("__exn_arg", Constructor(name, Some(Var("__exn_arg", unknownSpan)), unknownSpan), Map.empty)` -- BUT this won't work because Constructor creates DataValue not ExnValue. Instead, handle it by adding special-case logic:
     - For nullary: `env |> Map.add name (ExnValue(name, None))`
     - For with-data: Create a FunctionValue whose body when applied will produce ExnValue. The simplest approach: store the name in a closure and use a synthetic approach. Actually the cleanest approach: in the eval of Constructor, check if the constructor name maps to an exception (ResultType=TExn in ctorEnv). BUT eval doesn't have ctorEnv...
     - **Simplest approach**: Exception constructors with data are already parsed as `Constructor(name, Some arg)` by the parser (uppercase IDENT). In `eval`, the `Constructor` case creates `DataValue`. Instead, for exception constructors, we want `ExnValue`. The cleanest way: in `evalModuleDecls` for ExceptionDecl, register the constructor name in a set of exception names, and pass this set to eval. BUT this is invasive.
     - **Pragmatic approach**: Use `DataValue` for exception values at runtime. Change `ExnValue` approach -- DON'T use ExnValue. Instead, use `DataValue(name, valueOpt)` for exceptions too. The LangThreeException catches and pattern-matches on DataValue. This is simpler and reuses existing infrastructure. The type checker distinguishes exceptions from ADTs; the runtime doesn't need to.
     - **Decision: Use DataValue for exception runtime values.** Remove ExnValue from Ast.fs Value type (it was added in Plan 01). Exception constructors at runtime are just DataValue. This simplifies everything -- existing Constructor eval, matchPattern, and evalMatchClauses all work unchanged for exceptions.
   - With DataValue approach: ExceptionDecl handling in evalModuleDecls just needs to register the constructor in env (same as TypeDecl constructors). For nullary: `Map.add name (DataValue(name, None)) env`. For with-data: `Map.add name (FunctionValue("__x", Constructor(name, Some(Var("__x", unknownSpan)), unknownSpan), Map.empty)) env`.
   - Also register in moduleEnv.CtorEnv for qualified access.

**IMPORTANT SIMPLIFICATION**: Remove `ExnValue` from Ast.fs Value type. Exception values at runtime are represented as `DataValue(name, valueOpt)`. This means:
- matchPattern already handles `ConstructorPat` against `DataValue` correctly
- evalMatchClauses works unchanged for exception handlers
- No special runtime logic needed for exceptions vs ADTs
- Only the type checker distinguishes them (via TExn vs TData in ConstructorEnv)

Update Ast.fs to remove `| ExnValue of name: string * value: Value option` from the Value union.

  <action>
Add `InTry` syntax context to IndentFilter, modeled identically to `InMatch`:

1. Add `| InTry of baseColumn: int` to SyntaxContext union.

2. Add `JustSawTry: bool` field to FilterState (default false in initialState).

3. In `processNewlineWithContext`:
   - After the JustSawMatch check, add similar logic for JustSawTry:
     ```fsharp
     let stateWithTryContext =
         if stateWithMatchContext.JustSawTry then
             { stateWithMatchContext with Context = InTry stateWithMatchContext.IndentStack.Head :: stateWithMatchContext.Context; JustSawTry = false }
         else
             stateWithMatchContext
     ```
   - Use stateWithTryContext in place of stateWithMatchContext for the rest of the function.
   - In the DEDENT context popping, add: `| InTry baseCol :: rest when newState.IndentStack.Head <= baseCol -> popContexts rest`
   - In the pipe alignment check, also match `InTry baseCol :: _` (try-with handlers also use pipes at base column):
     ```fsharp
     | InTry baseCol :: _ when nextToken = Some Parser.PIPE ->
         if col <> baseCol then
             raise (IndentationError(...))
         (stateAfterIndent, [])
     ```

4. In `updateContextOnDedent`:
   - Add: `| InTry baseCol :: rest -> if state.IndentStack.Head <= baseCol then { state with Context = rest } else state`

5. In the `filter` function main loop:
   - Add a case for `Parser.TRY`: `| Parser.TRY -> state <- { state with JustSawTry = true; PrevToken = Some token }; yield token`
   - This mirrors the existing `Parser.MATCH` case.
