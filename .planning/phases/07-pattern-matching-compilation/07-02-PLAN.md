  <action>
Modify the `Match` case in the `eval` function (currently around line 170-172). Replace:

```fsharp
| Match (scrutinee, clauses, _) ->
    let value = eval recEnv moduleEnv env scrutinee
    evalMatchClauses recEnv moduleEnv env value clauses
```

With:

```fsharp
| Match (scrutinee, clauses, _) ->
    let value = eval recEnv moduleEnv env scrutinee
    let tree, rootVar = MatchCompile.compileMatch clauses
    let evalFn e expr = eval recEnv moduleEnv e expr
    let varEnv = Map.ofList [(rootVar, value)]
    MatchCompile.evalDecisionTree evalFn env varEnv tree
```

The `evalFn` closure captures `recEnv` and `moduleEnv` from the current scope. The rootVar maps to the scrutinee value in the initial varEnv.

**Do NOT change:**
- The `TryWith` case (lines ~423-433). Exception handlers must remain sequential via `evalMatchClauses` because exception types are open and the catch-all fallthrough with re-raise depends on sequential evaluation.
- The `LetPat` case (lines ~153-167). It uses `matchPattern` directly for single-pattern destructuring.

**Do NOT remove** `matchPattern` or `evalMatchClauses`. They are still used by TryWith and LetPat.

The only change is the Match case body (3 lines replaced with 5 lines).

**Debugging:** If any existing test fails after this change, the issue is in MatchCompile.fs (not in this wiring). Common causes:
- VarPat bindings lost during pushVarBindings or splitClauses
- Sub-pattern expansion in case (a) not mapping freshVars to sub-patterns correctly
- Record field sorting mismatch between extractSubPatterns and destructureValue
- When-guard fallthrough not compiled correctly (Leaf fallback missing)
- matchesConstructor not recognizing a value type

Fix any bugs found in MatchCompile.fs during this plan. The wiring itself is straightforward.
