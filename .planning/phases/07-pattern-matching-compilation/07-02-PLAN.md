<task type="auto">
  <name>Task 1: Wire decision tree compilation into Match expression evaluation</name>
  <files>src/LangThree/Eval.fs</files>
  <action>
Add `open MatchCompile` at the top of Eval.fs (after `open Type`).

Modify the `Match` case in the `eval` function (currently around line 170-172). Replace:

```fsharp
| Match (scrutinee, clauses, _) ->
    let value = eval recEnv moduleEnv env scrutinee
    evalMatchClauses recEnv moduleEnv env value clauses
```

With:

```fsharp
| Match (scrutinee, clauses, _) ->
    let value = eval recEnv moduleEnv env scrutinee
    let tree, bodies, guards = MatchCompile.compileMatch clauses
    let evalFn e expr = eval recEnv moduleEnv e expr
    MatchCompile.evalDecisionTree evalFn env value bodies guards tree
```

This passes `eval` as a first-class function to avoid circular module dependency. The `evalFn` closure captures `recEnv` and `moduleEnv` from the current scope.

**Do NOT change the TryWith case** (lines ~423-433). Exception handlers must remain sequential via `evalMatchClauses` because:
1. Exception types are open (new exceptions can be declared anywhere)
2. Exception handlers use the same MatchClause type but need sequential semantics for re-raising
3. The catch-all fallthrough with re-raise depends on sequential evaluation

**Do NOT change the LetPat case** (lines ~153-167). It uses `matchPattern` directly, not `evalMatchClauses`, and handles single patterns (not clause lists).

**Do NOT remove matchPattern or evalMatchClauses.** They are still used by:
- `evalMatchClauses`: TryWith exception handlers
- `matchPattern`: LetPat destructuring

**Keep all existing functions intact.** The only change is the Match case body (3 lines replaced with 4 lines).
