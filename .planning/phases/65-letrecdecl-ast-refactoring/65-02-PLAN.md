  <action>
Three key logic changes -- all follow the same pattern: instead of always creating a fresh type variable for paramTy, use `elaborateTypeExpr` when annotation is present.

IMPORTANT: All three files already have `open Elaborate` at the top, so `elaborateTypeExpr` is accessed as a bare function call (NOT `Bidir.elaborateTypeExpr` or `Elaborate.elaborateTypeExpr`). The function is defined in `Elaborate.fs` line 73.

When `elaborateTypeExpr` returns a concrete type (e.g., `TInt`), that type flows into the environment for the param binding. Downstream unification then correctly constrains the function body -- if the body uses the param in a way incompatible with the concrete type, unification fails with a type error. No additional unification call is needed; the existing `unify` calls in the type-check pipeline handle this naturally.

**TypeCheck.fs -- LetRecDecl (around line 852)**

Find the `funcTypes` binding where it maps over LetRecDecl bindings. Plan 01 updated the destructuring to include `paramTyOpt`. Now use it:
```fsharp
bindings |> List.map (fun (name, param, paramTyOpt, _body, _) ->
    let paramTy =
        match paramTyOpt with
        | Some tyExpr -> elaborateTypeExpr tyExpr
        | None -> Infer.freshVar()
```

**Bidir.fs -- LetRec (around line 302)**

Currently (after Plan 01 updated destructuring):
```fsharp
| LetRec (name, param, paramTyOpt, body, expr, span) ->
    let funcTy = freshVar()

```

Change `paramTy` binding:
```fsharp
| LetRec (name, param, paramTyOpt, body, expr, span) ->
    let funcTy = freshVar()

        match paramTyOpt with
        | Some tyExpr -> elaborateTypeExpr tyExpr
        | None -> freshVar()
```

        | None -> freshVar()
