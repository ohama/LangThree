<task type="auto">
  <name>Task 1: GADT constructor elaboration in Elaborate.fs</name>
  <files>src/LangThree/Elaborate.fs</files>
  <action>
Extend `elaborateTypeDecl` to handle `GadtConstructorDecl` in addition to `ConstructorDecl`.

For a type declaration like:
```
type 'a expr =
    | Int : int -> int expr
    | Bool : bool -> bool expr
    | If : bool expr * 'a expr * 'a expr -> 'a expr
```

The type has typeParams = ["'a"] which maps to paramMap = {"'a" -> 0}.

For each GadtConstructorDecl(name, argTypes, returnType, span):

1. **Elaborate argument types** using substTypeExprWithMap with paramMap. If argTypes is empty, ArgType = None. If argTypes has one element, ArgType = Some(elaborated type). If argTypes has multiple elements, ArgType = Some(TTuple(elaborated types)).

2. **Elaborate the return type** using substTypeExprWithMap with paramMap. This gives the explicit result type. For `Int : int -> int expr`, the return type `int expr` elaborates to TData("expr", [TInt]).

3. **Determine IsGadt**: A constructor is a GADT constructor if its return type instantiates type parameters to concrete types (not just TVar of the declared param). Compare the elaborated return type's args with the generic result type `TData(name, [TVar 0, TVar 1, ...])`. If they differ for ANY constructor in the type, the type is a GADT. Actually, simpler: a GadtConstructorDecl is always IsGadt=true (the syntax already distinguishes them).

   However, for the `If` constructor above, the return type is `'a expr` which is generic -- same as the declared type. But since it's declared with colon syntax alongside GADT constructors, it should still be marked IsGadt=true because the TYPE is a GADT type (other constructors have refined return types), and the match handling needs to know.

   Better approach: determine IsGadt at the TYPE level. After elaborating all constructors, check if ANY constructor has a return type that differs from the generic `TData(name, [TVar 0, TVar 1, ...])`. If so, mark ALL constructors of that type as IsGadt=true.

4. **Compute ExistentialVars**: For each GADT constructor, existential vars are type parameter indices that appear in the argument types but NOT in the return type's type arguments. For `Show : 'a * ('a -> string) -> showable`, if 'a has index 0 and it doesn't appear in the return type `showable` (which has no type args), then ExistentialVars = [0].

   Implementation: after elaborating, collect freeVars from the constructor's result type, then ExistentialVars = typeParamVars |> filter (not in resultType freeVars).

   Wait, there's a subtlety: GADT constructors may introduce type variables not in the type's declared params. For example:
   ```
   type showable =
       | Show : 'a * ('a -> string) -> showable
   ```
   Here `showable` has NO type params, but `Show` introduces 'a. We need to handle this.

   Approach: For GadtConstructorDecl, collect ALL type variable names from argTypes and returnType. For each, check if it's in the type's paramMap. If not, allocate a fresh type var index and add it to a local paramMap extension. These extended vars that don't appear in the return type are existential.

   Implementation steps for each GadtConstructorDecl:
   a. Start with the type's paramMap
   b. Collect all TEVar names from argTypes that are NOT in paramMap -- these are constructor-local type vars
   c. Allocate fresh indices for them, extend paramMap
   d. Elaborate argTypes and returnType with extended paramMap
   e. The constructor's TypeParams = type-level params ++ constructor-local params
   f. ExistentialVars = constructor-local param indices (vars NOT in return type freeVars)

5. **Set ResultType**: Use the elaborated return type, NOT the generic one. For `Int : int -> int expr`, ResultType = TData("expr", [TInt]). This is the key difference from regular ADTs where all constructors share the same generic ResultType.

The elaborateTypeDecl function signature doesn't change. It still returns ConstructorEnv. The ConstructorInfo records now use the extended fields.

For backward compatibility with regular ADT constructors (ConstructorDecl), keep them as-is but with IsGadt=false and ExistentialVars=[].

After building all constructor infos, do the IsGadt sweep: if any constructor in the type is from GadtConstructorDecl, mark all constructors as IsGadt=true.

1. The `TypeDeclaration` pattern in the Decls rule of Parser.fsy produces `Ast.TypeDecl` containing either `ConstructorDecl` or `GadtConstructorDecl` in the constructors list. The existing `Decl.TypeDecl` case in typeCheckModule extracts these and passes to `elaborateTypeDecl`. This should work as-is.

3. The `inferTypeFromPatterns` function currently gets constructor ResultType from ctorEnv. For GADT constructors, the ResultType is now the specific return type (e.g., TData("expr", [TInt]) for `Int`). This means the scrutinee type inference will find different result types for different constructors. The function should still work because it picks the FIRST constructor's result type -- but for GADTs, different constructors have different result types.

   Update `inferTypeFromPatterns` to handle GADT: instead of just returning the first constructor's ResultType, find the type NAME from any constructor and build the generic type. Specifically:
   ```fsharp
   let inferTypeFromPatterns (patterns: Pattern list) : Type option =
       patterns
       |> List.tryPick (fun pat ->
           match pat with
           | Ast.ConstructorPat(name, _, _) ->
               match Map.tryFind name ctorEnv with
               | Some info ->
                   match info.ResultType with
                   | TData(typeName, _) ->
                       // Build generic type for exhaustiveness (use type name)
                       Some (TData(typeName, info.TypeParams |> List.map TVar))
                   | other -> Some other
               | None -> None
           | _ -> None)
   ```
   This ensures exhaustiveness checking uses the generic type to find ALL constructors of that type, not just ones matching a specific instantiation.
