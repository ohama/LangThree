Add a new `TypeArgList` grammar rule (comma-separated, at least one type expression):

```fsharp
TypeArgList:
    | TypeExpr                      { [$1] }
    | TypeExpr COMMA TypeArgList    { $1 :: $3 }
```

Add two new alternatives to `AtomicType` (after the existing `LPAREN TypeExpr RPAREN` rule):

```fsharp
    | IDENT LT TypeArgList GT       { Ast.TEData($1, $3) }
```

Note: `Result<'a> list` composition works automatically because `AtomicType TYPE_LIST` already exists and will match `(IDENT LT TypeArgList GT) TYPE_LIST`. Similarly `Result<'a> option` works via `AtomicType IDENT`. No extra rules needed for mixed postfix/angle bracket.

Add the same angle bracket rule to `AliasAtomicType` (after the existing `LPAREN TypeExpr RPAREN` rule):

```fsharp
    | IDENT LT TypeArgList GT       { Ast.TEData($1, $3) }
```

IMPORTANT: Do NOT add separate LANGLE/RANGLE tokens. Reuse existing LT/GT tokens. The parser state machine disambiguates because type rules and expression rules are in disjoint LALR(1) states.

After adding, run `dotnet build src/LangThree/LangThree.fsproj -c Release` and check for shift/reduce conflict warnings. Zero warnings means the grammar is unambiguous. If conflicts appear on LT/GT, they need to be investigated (but research indicates this is unlikely).
