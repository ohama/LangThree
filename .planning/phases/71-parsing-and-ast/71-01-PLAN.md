   | InstanceDecl of className: string * instanceType: TypeExpr * methods: (string * Expr) list * Span
   ```

   ```

**Lexer.fsl — Add three new keyword tokens:**

1. Add these three rules in the keyword section (after the `"do"` rule, before the `'_'` wildcard rule):
   ```
   | "typeclass"   { TYPECLASS }
   | "instance"    { INSTANCE }
   ```

2. The `=>` (fat arrow) token: Add BEFORE the multi-char operator catch-all rule and BEFORE the `=` single-char rule. Place it near the other multi-char operators like `<=`, `>=`, `<>`:
   ```
   | "=>"          { FATARROW }
   ```
   Place this right after the `"<-"  { LARROW }` line (line ~129) since it's a multi-char operator starting with `=`.

**Parser.fsy — Token declarations and grammar rules:**

1. Add token declarations (after the loop keyword tokens section):
   ```
   // Phase 71 (Type Classes): typeclass/instance tokens
   %token TYPECLASS INSTANCE FATARROW
   ```

2. Add typeclass declaration rules in the `Decls` nonterminal (after the LetRecDeclaration rules, around line 672). Follow the existing pattern where each declaration form has two variants (standalone and followed by more Decls):
   ```
   // Phase 71 (Type Classes): typeclass declaration
   | TYPECLASS IDENT TYPE_VAR EQUALS INDENT TypeClassMethodList DEDENT
       { [TypeClassDecl($2, $3, $6, ruleSpan parseState 1 7)] }
   | TYPECLASS IDENT TYPE_VAR EQUALS INDENT TypeClassMethodList DEDENT Decls
       { TypeClassDecl($2, $3, $6, ruleSpan parseState 1 7) :: $8 }
   // Phase 71 (Type Classes): instance declaration
   | INSTANCE IDENT AtomicType EQUALS INDENT InstanceMethodList DEDENT
       { [InstanceDecl($2, $3, $6, ruleSpan parseState 1 7)] }
   | INSTANCE IDENT AtomicType EQUALS INDENT InstanceMethodList DEDENT Decls
       { InstanceDecl($2, $3, $6, ruleSpan parseState 1 7) :: $8 }
   ```

3. Add helper nonterminals (place them after the MixedParamList rules, around line 760):

   ```
   // Phase 71 (Type Classes): typeclass method signatures
   TypeClassMethodList:
       | TypeClassMethod                           { [$1] }
       | TypeClassMethod TypeClassMethodList        { $1 :: $2 }

   TypeClassMethod:
       | IDENT COLON TypeExpr                      { ($1, $3) }

   // Phase 71 (Type Classes): instance method definitions
   InstanceMethodList:
       | InstanceMethod                            { [$1] }
       | InstanceMethod InstanceMethodList          { $1 :: $2 }

   InstanceMethod:
       | LET IDENT ParamList EQUALS SeqExpr
           { let lambda = List.foldBack (fun param body -> Lambda(param, body, ruleSpan parseState 1 5)) $3 $5
             ($2, lambda) }
       | LET IDENT ParamList EQUALS INDENT SeqExpr DEDENT
           { let lambda = List.foldBack (fun param body -> Lambda(param, body, ruleSpan parseState 1 7)) $3 $6
             ($2, lambda) }
   ```

4. Add constrained type expression rule. Modify the `TypeExpr` nonterminal to support `ClassName TypeVar => TypeExpr`:
   ```
   TypeExpr:
       | ConstraintList FATARROW ArrowType  { TEConstrained($1, $3) }
       | ArrowType                          { $1 }

   ConstraintList:
       | Constraint                         { [$1] }
       | Constraint COMMA ConstraintList    { $1 :: $3 }

   Constraint:
       | IDENT TYPE_VAR                     { ($1, TEVar($2)) }
   ```
   Replace the existing single-rule `TypeExpr` (which is just `| ArrowType { $1 }`) with the two-rule version above.

IMPORTANT LALR(1) note: The `ConstraintList` uses `IDENT TYPE_VAR` which could conflict with `AtomicType` since `IDENT` alone is `TEName`. The key disambiguation is that `ConstraintList` requires `IDENT TYPE_VAR FATARROW` — the lookahead to `FATARROW` after `TYPE_VAR` resolves the ambiguity. If fsyacc reports a shift/reduce conflict, wrap the constraint in parentheses instead: `(Show 'a) => ...` and use `LPAREN IDENT TYPE_VAR RPAREN` in the Constraint rule. Try the simpler version first.
