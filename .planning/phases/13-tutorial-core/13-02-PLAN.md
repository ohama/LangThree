<task id="1" title="Chapter 5: Algebraic Data Types">
Write `tutorial/05-algebraic-types.md`:

**Content to cover:**
- Simple enumerations: `type Color = Red | Green | Blue`
- Constructors with data: `type Shape = Circle of int | Rect of int * int`
- Parametric types: `type Option 'a = None | Some of 'a` (in Prelude)
- Recursive types: `type IntList = Nil | Cons of int * IntList`
- Mutually recursive types: `type A = ... and B = ...`
- Pattern matching on ADTs (link to ch4)
- Exhaustiveness: compiler warns if you miss a constructor
- Leading pipe syntax: `type T = | A | B` (optional leading pipe)

**LangThree-specific syntax:**
- Type parameters AFTER type name: `type Option 'a` (not `'a Option`)
- Constructor matching: `match opt with | Some x -> x | None -> 0`
- Type parameter in expressions: appears as `Option<int>` in type output

**Examples:**
- Define and use Option (from Prelude)
- Binary tree type with depth function (using inline let rec)
- Expression type for simple calculator
</task>

<task id="2" title="Chapter 6: Records">
Write `tutorial/06-records.md`:

**Content to cover:**
- Record type declaration: `type Point = { x: int; y: int }`
- Record creation: `let p = { x = 1; y = 2 }`
- Field access (dot notation): `p.x`
- Chained field access: `outer.inner.value`
- Copy-and-update: `{ p with x = 10 }`
- Multi-field update: `{ p with x = 10; y = 20 }`
- Record pattern matching: `match p with | { x = a; y = b } -> a + b`
- Mutable fields: `type Counter = { mutable count: int }`
- Mutable update: `c.count <- c.count + 1` (returns unit)
- Parametric records: `type Pair 'a = { first: 'a; second: 'a }`
- Structural equality: `p1 = p2` compares field values

**LangThree-specific:**
- Field names must be globally unique across all record types
- `<-` for mutable assignment (returns unit, use `let _ =` for sequencing)
- No record field punning
</task>

<task id="3" title="Chapter 7: GADTs">
Write `tutorial/07-gadt.md`:

**Content to cover:**
- What GADTs are and why (brief — audience knows FP)
- GADT constructor syntax: `type Expr 'a = IntLit : int -> int Expr | BoolLit : bool -> bool Expr`
- Difference from regular ADT: explicit return type per constructor
- Type refinement in pattern matching:
  ```
  let eval e = (match e with | IntLit n -> n | BoolLit b -> 0 : int)
  ```
- **Critical:** annotation `: int` at end of match expression enters check mode
- Adding more constructors: `Add : int Expr * int Expr -> int Expr`
- Existential types (brief mention)
- GADT exhaustiveness: compiler correctly filters impossible constructors
- Common error: E0401 for unannotated GADT match (with explanation)

**LangThree-specific syntax:**
- Type param after name: `type Expr 'a = ...`
- Return type uses `int Expr` (not `Expr int` or `Expr<int>`)
- Match annotation: `(match e with | ... : ResultType)` — the `: Type` goes INSIDE the parens at the end
- If any constructor uses GADT syntax, ALL constructors are treated as GADT

**Examples:**
- Type-safe expression evaluator (IntLit, BoolLit, Add)
- Show what happens without annotation (E0401 error)
- Show type refinement: `IntLit n -> n + 1` works because n is known to be int
</task>
